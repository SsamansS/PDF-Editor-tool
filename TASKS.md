# TASKS.md — Трекер задач PDF Editor Tool

Единый источник правды по всем задачам проекта. Любая работа над проектом начинается
с открытия этого файла и заканчивается обновлением статуса здесь.

---

## Как трекается исполнение

**Источник правды — этот файл.** Перед началом задачи открываем `TASKS.md`,
после завершения — обновляем статус и пишем строку в «Журнал».

### Статусы
| Значок | Статус | Значение |
|--------|--------|----------|
| ☐ | TODO | не начато |
| 🔄 | IN PROGRESS | в работе (одновременно держим максимум одну задачу) |
| ✅ | DONE | сделано **и проверено** по критерию готовности (DoD) |
| ⛔ | BLOCKED | заблокировано — причина рядом |
| ❌ | DROPPED | решили не делать — причина в Журнале |

### Правила ведения
1. Задачи сгруппированы в **эпики** (`E1`, `E2`, …). У каждой задачи свой **ID** (`T1`, `T1.2`, …) — уникальный в пределах файла.
2. На ID ссылаемся в коммитах: `[T2] add upload endpoint`.
3. Статус меняем синхронно: в **таблице-обзоре эпика** и в заголовке самой задачи.
4. Задача → `✅ DONE` только когда выполнен её **DoD (Definition of Done)**, а не когда «код написан».
5. Каждое изменение статуса фиксируем строкой в разделе **Журнал выполнения** (дату беру из системы на момент записи).
6. Активную задачу дублирую во встроенный трекер Claude Code (TaskCreate/TaskUpdate) — для оперативной видимости в сессии. `TASKS.md` = долговременная память, трекер = оперативная.
7. Новые задачи, всплывшие по ходу, добавляем сюда с новым ID — не держим в голове.
8. Новый крупный блок работ = новый эпик (`E2`, `E3`, …) со своей таблицей-обзором.

# E1 — Публикация в вебе на поддомене `pdf.runpro.in`

Цель: коллеги пользуются инструментом через браузер по адресу **`https://pdf.runpro.in`** —
безопасно, с HTTPS, не затрагивая существующий сайт и API на этом же сервере.

> **Решение по способу публикации: Вариант A — поддомен** (подтверждено 2026-06-23).
> Домен `runpro.in` с валидным SSL уже есть на сервере — покупать ничего не нужно.
> Подпуть `runpro.in/pdf/` отклонён: наш app использует абсолютные пути `/api/`, `/static/`,
> а `/api/` на nginx уже занят Java-бэкендом → конфликт. Поддомен даёт чистое разделение.

> Контекст приложения: сейчас оно локальное. Сервер слушает `127.0.0.1:8765`
> (`pdf_editor_web.py:437`), файлы открываются по **пути на диске** (`_resolve_pdf`, поле
> «Полный путь к PDF…» в `index.html:13`), авторизации нет, бэкенд — встроенный `http.server`.
> «Выложить как есть» нельзя: коллега не сможет выбрать свой файл, и доступ будет открыт всем.

## Факты о сервере (аудит T0, 2026-06-23)
- **Хост:** `ubuntu@13.59.170.121`, Ubuntu 24.04, 2 CPU, 7.6 ГБ RAM (6.4 свободно), диск 14 ГБ свободно.
- **nginx** (80, 443): обслуживает `runpro.in` / `www.runpro.in` с SSL Let's Encrypt (Certbot).
  `/` → SPA из `/var/www/runpro`; `/api/`, `/auth/` → proxy на `localhost:8080`.
  Конфиг: `/etc/nginx/sites-available/runpro` (симлинк в `sites-enabled`).
- **Spring Boot** `bot.jar` (8080): backend, сервис `runpro-bot.service`, путь `/opt/runpro/backend`.
- **PostgreSQL 16** (5432, localhost) · **MinIO** (9000/9001, localhost).
- **Python 3.12** системный. **Свободный порт для нас: `8765`.**
- ⚠️ Неинтерактивный SSH имеет урезанный `PATH` — в командах задавать
  `export PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin`.

## 🚧 Guardrails — что НЕЛЬЗЯ трогать (нарушит работающий сайт/API)
- **Порты** 80, 443, 8080, 9000, 9001, 5432, 22 — не занимать, не переназначать.
- **Сервисы** `nginx`, `runpro-bot`, `postgresql@16-main`, `minio` — не останавливать/менять.
- **nginx-конфиг**: только **добавить новый** `server`-блок для `pdf.runpro.in`; блок `runpro.in`
  (его `/`, `/api/`, `/auth/`, ssl) **не редактировать**. После правок — `sudo nginx -t` перед reload.
- **Наш Python-процесс** слушает **только `127.0.0.1:8765`** (за nginx), НЕ `0.0.0.0`.
- Каталоги `/var/www/runpro` и `/opt/runpro/backend` — не трогать. Наш код — в отдельном каталоге.
- Новый systemd-юнит с уникальным именем `pdf-editor.service`.
- В Security Group **новые порты НЕ открывать** (80/443 уже открыты, трафик идёт через nginx).

### B1 — Правки части PDF не применяются 🟠 ✅ DONE
**Симптом:** при редактировании некоторых блоков PDF изменение не отображается (правится «не тот» блок).
**Воспроизведение:** бьёт по блокам, чей текст (или изменённый фрагмент) дублируется **выше** на той же странице. Уникальные блоки правятся нормально → отсюда «в каких-то областях работает, в каких-то нет».
**Причина (установлена 2026-06-27):** в `pdf_core.py` функция `replace_block` сначала пробует
замену в content-stream — `_content_stream_replace` → `_replace_in_pdf_strings` (строки 592-685).
Этот поиск **глобален по всей странице и игнорирует `bbox` блока**, заменяет **первое** совпадение
(`limit=1`, вызовы на строках 1116 и 1143). Если такой же текст встречается на странице раньше —
правка уходит в чужой блок, рапортуется `changed=True`, а выбранный блок остаётся прежним.
Стратегия 1b (difflib-фрагмент, 1121-1146) усугубляет: меняет первый совпавший фрагмент по странице.
Корректная bbox-привязанная Стратегия 3 (redact+insert, 1162-1180) не достигается, т.к. Стратегия 1
раньше вернула «успех».
**Фикс (план):** перед Стратегиями 1/1b проверять число вхождений на странице
(`len(page.search_for(old_single)) > 1`) → при дубле пропускать глобальные стратегии и идти к
bbox-привязанным (2/3). Для уникального текста поведение не меняется.
**Фикс (сделано 2026-06-27, ветка `fix_bugs`):** в `replace_block` добавлен гард `_occurs_more_than_once`
(через `page.search_for`): глобальные content-stream стратегии 1/1b применяются только если текст
(или фрагмент) на странице уникален; при дубле — сразу bbox-привязанные стратегии 2/3.
**DoD:** ✅ правка дублирующегося блока меняет именно выбранный блок (а не первый на странице);
проверено before/after на PDF с двумя одинаковыми блоками (старый код менял верхний — новый меняет
целевой); уникальные блоки правятся как прежде (регрессии нет).
